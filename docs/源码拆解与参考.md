> 目的：调研公开开源的 Discord 本地客户端翻译插件，提炼可借鉴的架构、UI 注入方式、翻译调用和错误处理策略。  
> 原则：只参考公开源码，不处理闭源/私有插件，不复制 GPL 代码到本项目。

## 2. 许可证结论

### 2.1 Vencord

Vencord 使用 GPL-3.0-or-later。其官方 Translate 插件也带 GPL 头。

影响：

- 可以阅读、学习、对照思想。
- 如果直接复制代码，衍生项目可能需要遵循 GPL-3.0-or-later。
- 本项目如未来想保持自用或非 GPL 发布，应避免直接复制实现代码。

### 2.2 BetterDiscordAddons

BetterDiscordAddons 使用 GPL-2.0。

影响：

- 可以阅读和学习实现方式。
- 不能直接搬运核心函数到本项目，除非接受 GPL 约束。
- 可重新实现同类思想：prompt 设计、错误码分类、备用引擎思路。

### 2.3 本项目原则

本项目应采用“clean-room 风格参考”：

- 记录公开插件的架构思路。
- 不逐行复制源码。
- 自己实现消息识别、翻译服务、缓存、UI 样式。
- 对外发布前再次审查许可证。

## 3. Vencord Translate 插件拆解

### 3.1 核心定位

Vencord 官方 Translate 插件支持：

- 右键消息菜单翻译。
- 消息 hover popover 按钮翻译。
- 消息下方 accessory 显示译文。
- 聊天输入栏按钮打开翻译设置。
- 自动翻译“自己将要发送的消息”。
- 翻译服务支持 Google Translate、DeepL、Kagi。

注意：它的 `autoTranslate` 主要是对**发送消息**生效：`onBeforeMessageSend` 中把用户将发送的内容翻译后再发送。它不是我们想要的“自动翻译当前可见收到消息”。

### 3.2 `index.tsx` 的插件入口

关键能力：

```text
contextMenus.message
messagePopoverButton
renderMessageAccessory
chatBarButton
onBeforeMessageSend
```

可借鉴点：

1. **消息内容提取**
   - 通过 Discord `Message` 对象读取 `message.content`。
   - 兼容 `messageSnapshots`。
   - 兼容部分 embed 描述。

2. **右键菜单插入**
   - 在消息右键菜单的 `copy-text` 后插入 `Translate` 菜单项。
   - 点击后调用 `translate("received", content)`。
   - 翻译结果用 `handleTranslate(message.id, trans)` 写入 accessory。

3. **悬浮按钮插入**
   - `messagePopoverButton` 返回翻译按钮。
   - 点击按钮后翻译对应消息。

4. **消息下方渲染**
   - `renderMessageAccessory: props => <TranslationAccessory message={props.message} />`
   - 这是本项目最值得借鉴的点：Vencord 提供了消息 accessory 渲染入口，比自己 DOM append 更稳定。

5. **发送前翻译**
   - `onBeforeMessageSend` 对用户将发送的消息翻译后替换。
   - 本项目不做发送消息翻译，可不实现。

### 3.3 `TranslationAccessory.tsx` 的译文显示机制

核心机制：

```text
TranslationSetters: Map<messageId, setState>
handleTranslate(messageId, data) → 找到对应 message 的 setState → 设置译文
TranslationAccessory(message) → message 下方显示译文
```

可借鉴点：

- 以 `message.id` 作为译文状态映射 key。
- 每条消息 accessory 内部维护自己的 React state。
- 没有译文时返回 `null`，不影响原消息 UI。
- 有译文时显示一个低调 accessory。
- 提供 `Dismiss` 按钮清除译文。

本项目建议：

- 继续采用 `messageId → setter` 的思路。
- 增加缓存层：`messageId/contentHash → translation`。
- 自动翻译时，不需要用户点击，消息 accessory mount 后即可入队翻译。
- 译文显示样式改为中文阅读友好，不用斜体英文风格。

### 3.4 `settings.tsx` 和 `TranslateModal.tsx`

Vencord 官方插件设置项：

- receivedInput
- receivedOutput
- sentInput
- sentOutput
- service
- deeplApiKey
- kagiSession
- autoTranslate
- showAutoTranslateTooltip
- manageTranslateSettings

可借鉴点：

- 复杂设置放 modal。
- 基础开关放插件 settings。
- 服务商配置根据选择动态隐藏/显示。

本项目建议设置：

```text
autoTranslateVisibleMessages: boolean
translationServiceUrl: string，默认 http://127.0.0.1:3789
minEnglishChars: number，默认 8
maxChars: number，默认 2000
concurrency: number，默认 6
cacheEnabled: boolean
showOriginal: boolean，可选
```

API Key 不进入 Vencord 设置页，只由本地服务读取 `DEEPSEEK_API_KEY`。

### 3.5 `utils.ts` 的翻译服务抽象

Vencord 官方插件的 `translate(kind, text)` 会根据设置选择：

- Google Translate
- DeepL
- Kagi

并统一返回：

```ts
interface TranslationValue {
  sourceLanguage: string;
  text: string;
}
```

可借鉴点：

- 插件层统一只关心 `TranslationValue`。
- 不同翻译服务细节隐藏在工具函数里。
- 错误用 toast 提示。
- DeepL/Kagi 走 `native.ts` 绕过浏览器 CORS 限制。

本项目建议：

- 插件层不直连 DeepSeek 官方 API。
- 插件层只调用本地服务：`POST http://127.0.0.1:3789/translate`。
- 本地服务返回统一结构：

```json
{
  "ok": true,
  "translated": "...",
  "cached": false,
  "source": "deepseek-v4-flash"
}
```

### 3.6 `native.ts` 的意义

Vencord 官方插件将 DeepL/Kagi 请求放在 `native.ts` 中执行，因为浏览器环境会遇到 CORS/权限问题。

本项目有两种选择：

1. 使用 `native.ts` 直接调用本地服务或 DeepSeek。
2. 更简单：`index.tsx` 浏览器侧直接 `fetch("http://127.0.0.1:3789/translate")`。

建议先用方案 2。如果 Discord/Vencord 环境限制 localhost fetch，再考虑 native 层。

### 3.7 `styles.css`

Vencord 官方译文样式：

```css
.vc-trans-accessory {
  color: var(--text-muted);
  margin-top: 0.5em;
  font-style: italic;
  font-weight: 400;
  line-height: 1.2rem;
  white-space: break-spaces;
}
```

本项目建议：

- 不使用 italic，中文长句斜体不舒服。
- 使用浅色左边框或轻微背景。
- 保持低干扰，不像机器人回复。

示意：

```text
译：你们能继续更新 SWE 模型吗？
```

## 4. BetterDiscord Translator 拆解

### 4.1 核心定位

BetterDiscord Translator 更“全能”：

- 翻译收到消息。
- 翻译发出消息。
- 支持多个翻译引擎。
- 支持备用翻译引擎。
- 支持前缀触发特定语言翻译，例如 `$fr Hello world`。
- 支持禁用翻译前缀。
- 支持 per-chat translation。

### 4.2 重要功能点

README 记录的两个设计值得参考：

1. **Translation Prefixes**
   - `$fr Hello world` → 翻译成法语。
   - `$de How are you?` → 翻译成德语。

2. **Disable Translation Prefixes**
   - 在自动翻译开启时，通过特殊前缀让某条消息不翻译。

本项目当前不做发送消息翻译，但“禁用翻译前缀/跳过规则”可参考为：

- 纯代码块跳过。
- 以指定前缀开头的消息跳过。
- 用户名、命令、URL 保留。

### 4.3 多引擎和备用引擎

BetterDiscord Translator 有 `validTranslator`、主引擎、备用引擎的逻辑：

```text
主翻译引擎失败 → 如果备用引擎可用 → 使用备用引擎
```

可借鉴点：

- 错误不要直接崩。
- 401/403：API key 错误。
- 429：额度/频率限制。
- 其他：服务不可用。
- 失败时可以降级或显示简短状态。

本项目 MVP 暂不做多后端，但要保留接口边界。

### 4.4 DeepSeek 翻译逻辑

BetterDiscord Translator 已经包含 `deepSeekTranslate(data, callback)`，核心策略：

- 模型：`deepseek-v4-flash`。
- 端点：`https://api.deepseek.com/v1/chat/completions`。
- 请求格式：OpenAI compatible。
- Prompt 定位：professional localization expert。
- system：senior bilingual localization specialist。
- 参数：`temperature: 0.2`，`top_p: 0.8`。
- 通过 `[NEWLINE]` 标记保留换行，翻译后再还原为 `\n`。
- 错误处理：
  - 401/403 → key 错误或过期。
  - 429 → 请求限额。
  - 其他 → 服务异常。

可借鉴点：

- `deepseek-v4-flash` 适合翻译。
- 低温度参数适合稳定翻译。
- `[NEWLINE]` 技巧可保留换行结构。
- 错误码分类很有用。

不能直接复制的点：

- 不能把 API Key 放进插件设置或客户端请求里。
- 不直接复制它的函数实现。
- 不需要 BetterDiscord 的 BDFDB 依赖结构。

本项目应把 DeepSeek 调用放到本地 `translation-service`。

## 5. 对本项目的直接启发

### 5.1 不需要从零摸 Discord 消息渲染

Vencord 已经提供：

- `renderMessageAccessory`
- `messagePopoverButton`
- `contextMenus.message`

所以第一版不应该直接用脆弱 DOM selector。优先尝试 Vencord 的消息渲染入口。

### 5.2 自动翻译可见消息的实现思路

Vencord 官方 Translate 没做“自动翻译收到的可见消息”，但它的 accessory 模型可扩展：

```text
TranslationAccessory mount
    ↓
拿到 message.id + message.content
    ↓
过滤是否需要翻译
    ↓
查缓存
    ↓
入队翻译
    ↓
setTranslation
    ↓
显示译文
```

这比 MutationObserver 更稳。

如果 `renderMessageAccessory` 对所有消息都会 mount，那么我们可以在 accessory 组件内部触发自动翻译。

### 5.3 推荐状态机

每条消息翻译状态：

```text
idle → queued → translating → translated
                 ↓
                failed
```

状态数据：

```ts
type TranslationState =
  | { status: "idle" }
  | { status: "queued" }
  | { status: "translating" }
  | { status: "translated"; text: string; cached: boolean }
  | { status: "failed"; error: string };
```

### 5.4 推荐模块拆分

```text
plugin/
├── index.tsx                 # Vencord 插件入口
├── TranslationAccessory.tsx  # 每条消息下方译文组件
├── settings.tsx              # 插件设置
├── styles.css                # UI 样式
├── translateClient.ts        # 调 localhost 服务
├── messageFilter.ts          # 是否需要翻译
├── cache.ts                  # 插件侧轻量缓存/状态
└── queue.ts                  # 限流队列

service/
├── server.mjs                # 本地 HTTP 服务
├── deepseek-client.mjs       # DeepSeek 官方 API
├── cache.mjs                 # 文件/内存缓存
├── config.mjs                # 配置读取
└── prompts.mjs               # 翻译 prompt
```

## 6. 本项目 MVP 更新建议

基于源码调研，MVP 应改成：

1. 先验证 Vencord `renderMessageAccessory` 是否能拿到当前消息。
2. 在 accessory 中显示静态测试中文。
3. 加入过滤规则，只对英文消息显示“翻译中”。
4. 接入本地 `translation-service` mock。
5. 接入 DeepSeek 官方 API。
6. 加入队列、缓存、限流。
7. 最后才做设置页和频道级开关。

原因：最核心风险不是 DeepSeek API，而是 Discord 消息组件注入点是否稳定。

## 7. 与现成插件的差异化

现成插件已有：

- 手动翻译按钮。
- 右键翻译。
- 多翻译服务。
- 发送前翻译。

本项目差异化：

- 默认面向中文开发者阅读英文 AI 社群。
- 自动翻译当前可见收到消息。
- 使用 DeepSeek 官方 API 做技术语境翻译。
- API Key 隔离在本地服务，不放插件。
- 后续可沉淀 Discord AI 社群情报，但第一版不做 Obsidian 联动。

## 9. 结论

结论非常明确：

1. 网上已有成熟 Discord 翻译插件。
2. Vencord 官方 Translate 插件是最佳结构参考。
3. BetterDiscord Translator 是 DeepSeek/OpenAI-compatible 翻译调用参考。
4. 两者都有 GPL 许可证，不能直接复制源码。
5. 我们应该复用思路而不是代码：
   - Vencord 的 `renderMessageAccessory` 思路。
   - `messageId → translation state` 思路。
   - DeepSeek 低温 prompt 翻译思路。
   - 错误码分类和缓存限流思路。
6. 第一开发重点应从“写翻译服务”调整为“验证 Vencord accessory 注入是否稳定”。
