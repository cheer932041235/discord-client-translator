## 3. 插件基座调研结论

### 3.1 Vencord

官方文档要点：

- 需要 development Vencord install。
- 推荐自定义插件放在 `src/userplugins`。
- `src/plugins` 用于官方插件，不适合个人私有插件。
- 插件可写成单文件 `myPlugin.ts` / `myPlugin.tsx`，也可写成文件夹 `myPlugin/index.tsx`。
- 插件入口是 `index.ts` 或 `index.tsx`。
- `index.ts(x)` 运行在浏览器环境，可使用浏览器 API。
- `index.ts(x)` 不能直接使用 Node.js 的 `fs` / `child_process`。
- 如果需要 Node.js API，可使用 `native.ts`。
- 插件添加后需要 rebuild Vencord，然后重启 Discord。

结论：Vencord 更适合作为第一候选，因为它明确支持私有 userplugins、TypeScript/React 开发和较现代的插件结构。

### 3.2 BetterDiscord

官方文档要点：

- 插件开发文档完整，涵盖 Quick Start、Plugin Structure、Environment、Developer Tools。
- 提供 DOM、设置页、React、UI 组件等教程。
- 开发门槛较低，适合快速 DOM 注入型插件。

结论：BetterDiscord 可作为备选方案。如果 Vencord 插件开发/安装流程过重，或者消息组件 hook 难度高，可以用 BetterDiscord 做快速 MVP。

### 3.3 当前推荐

第一阶段优先路线：

```text
Vencord userplugin + 本地 Node.js translation-service + DeepSeek 官方 API
```

BetterDiscord 暂作备选，不作为第一实现目标。

## 4. MVP 目标

MVP 不做按钮优先，而是直接做自动翻译可见消息。

### 必须实现

- 插件启动后监听当前频道消息列表。
- 识别当前可见英文消息。
- 自动发起翻译请求。
- 在原消息下方插入中文译文。
- 对同一消息去重，不重复翻译。
- 本地缓存翻译结果。
- 失败时不影响 Discord 原生 UI。
- 提供全局开关：启用/暂停自动翻译。

### 暂不实现

- Obsidian 收藏。
- 频道摘要。
- 右键菜单翻译。
- 手动发消息或机器人功能。
- 历史消息批量抓取。
- 多 API 后端切换。

## 6. 本地翻译服务接口草案

### 6.1 健康检查

```http
GET /health
```

响应：

```json
{
  "ok": true,
  "model": "deepseek-v4-flash"
}
```

### 6.2 翻译接口

```http
POST /translate
Content-Type: application/json
```

请求：

```json
{
  "messageId": "optional-discord-message-id",
  "channelId": "optional-channel-id",
  "text": "Can you guys please keep updating the SWE model?",
  "mode": "auto"
}
```

响应：

```json
{
  "ok": true,
  "translated": "你们能继续更新 SWE 模型吗？",
  "cached": false
}
```

失败响应：

```json
{
  "ok": false,
  "error": "DeepSeek API timeout"
}
```

## 8. 缓存策略

第一版允许本地缓存，但需要控制边界。

### MVP 缓存

- 内存缓存：默认开启。
- 文件缓存：允许开发，但必须可关闭、可清理。

### 缓存 Key

优先：

```text
channelId + messageId + contentHash
```

兜底：

```text
textHash
```

### 缓存内容

```json
{
  "hash": "...",
  "translated": "...",
  "createdAt": "2026-05-20T05:20:00+08:00"
}
```

缓存不保存 token、cookie、账号信息。

## 9. 自动翻译策略

为了避免 API 爆量，自动翻译不是“所有 DOM 变化都立刻请求”。

建议策略：

1. MutationObserver 监听消息列表变化。
2. 用 debounce 合并短时间变化。
3. 每次只处理视口附近消息。
4. 过滤中文、短文本、链接、emoji、代码块。
5. 并发限制为 2。
6. 对队列做去重。
7. 翻译失败重试最多 1 次。
8. 滚动时优先翻译最新可见消息。
